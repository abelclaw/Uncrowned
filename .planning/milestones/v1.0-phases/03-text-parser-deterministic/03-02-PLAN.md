  <action>
    **1. Update room JSONs with text content:**

    Add to each room JSON:
    - Top-level `description` field with a 1-2 sentence room description (e.g. "You stand in a sun-dappled forest clearing. A gnarled old stump sits among the ferns.")
    - `responses` map on each hotspot with at minimum `look` and `default` responses. Example for "old-stump": `{ "look": "A weathered tree stump, its rings telling centuries of stories. Something glints in a crack.", "take": "The stump is far too heavy to move.", "default": "You poke at the old stump. Nothing interesting happens." }`
    - `direction` and `label` fields on each exit. Example: forest_clearing's "to-cave" exit gets `"direction": "east", "label": "cave"`. cave_entrance's "to-forest" gets `"direction": "west", "label": "forest"`, "to-village" gets `"direction": "east", "label": "village"`. village_path's "to-cave" gets `"direction": "west", "label": "cave"`.

    Write evocative, in-character descriptions. The narrator voice is sardonic dark comedy (think Stanley Parable / Sierra death screens). Keep descriptions concise (1-3 sentences).

    **2. Create TextInputBar (src/game/ui/TextInputBar.ts):**

    HTML-based text input bar. Constructor takes the game container HTMLElement.

    Creates a wrapper div `#text-parser-ui` containing:
    - `#parser-response` div for showing command responses (plain text, no typewriter -- Phase 4 adds that)
    - `#parser-input` input element with placeholder "What do you do?"

    Features:
    - Enter key submits: gets trimmed value, pushes to history array, resets historyIndex, clears input, emits `EventBus.emit('command-submitted', text)`
    - Up/down arrows navigate command history (up = older, down = newer, past end = empty)
    - `showResponse(text: string)` method updates responseEl.textContent
    - `show()` / `hide()` methods toggle visibility (for scene transitions)
    - `focus()` method to give focus to input after scene is ready
    - `destroy()` method removes the wrapper from DOM

    Do NOT steal focus from canvas on pointer events. The input should auto-focus when the scene is ready. Pointer events on the canvas (click-to-move) should NOT call inputEl.blur() or prevent default on the input.

    **3. Create CommandDispatcher (src/game/systems/CommandDispatcher.ts):**

    Takes a GameAction and RoomData, returns a response string.

    `dispatch(action: GameAction, roomData: RoomData): { response: string, handled: boolean }`

    Verb handlers:
    - `look`: No subject -> return roomData.description. Subject is hotspot -> return hotspot.responses.look or "Nothing special about the {name}." Subject is exit -> return "You see a path leading {direction}." Subject unresolved -> "You don't see that here."
    - `go`: Subject is exit (by direction, label, or ID) -> emit `EventBus.emit('go-command', exit)` and return "You head {direction}." Subject is direction with no matching exit -> "You can't go that way." Subject unresolved -> "Go where?"
    - `take`: Subject is hotspot -> return hotspot.responses.take or "You can't take the {name}." Subject unresolved -> "You don't see that here." (Phase 4 extends with inventory.)
    - `use`: Subject is hotspot -> return hotspot.responses.use or "You can't figure out how to use that." Two-noun -> "Using {subject} on {target} doesn't seem to do anything." (Phase 4 extends.)
    - `talk`: Subject is hotspot -> return hotspot.responses.talk or "You talk to the {name}. It doesn't respond." (Phase 6 extends for NPCs.)
    - `open`: Subject is hotspot with target -> return hotspot.responses.open or "It doesn't open." Push/pull similarly.
    - `push`/`pull`: Subject is hotspot -> return hotspot.responses.push/pull or "Nothing happens."
    - Unrecognized/no match: return handled:false with a sardonic narrator response: "I don't understand what you mean by '{rawInput}'. Try something like 'look at stump' or 'go east'."

    **4. Wire into RoomScene (src/game/scenes/RoomScene.ts):**

    Import TextParser, CommandDispatcher, TextInputBar.

    In create():
    - After existing setup, create TextInputBar instance (pass game-container element: `document.getElementById('game-container')!`)
    - Create CommandDispatcher instance
    - Create TextParser instance
    - Listen on EventBus for 'command-submitted': when received, call TextParser.parse(input, roomData.hotspots, roomData.exits), then CommandDispatcher.dispatch(action, roomData), then textInputBar.showResponse(response)
    - Listen on EventBus for 'go-command': when received, call handleExitReached(exit) to trigger scene transition (reuse existing method)
    - After transition-in effect completes, call textInputBar.focus()

    In shutdown cleanup:
    - textInputBar.destroy()
    - Remove EventBus listeners for 'command-submitted' and 'go-command'

    IMPORTANT: The TextInputBar must persist across scene restarts. Two approaches:
    - Option A: Destroy and recreate on each scene create() (simpler, slight flicker)
    - Option B: Create once and reuse (complex lifecycle management)
    Use Option A for simplicity. The input bar is recreated each time RoomScene.create() runs. The destroy() in shutdown ensures no duplicates.

    IMPORTANT: Click-to-move must coexist. The pointerdown handler on the canvas should NOT blur the input. Do not call `e.stopPropagation()` on canvas pointer events in a way that breaks the input. The input bar is below the canvas in layout, not overlapping.

    **5. Update CSS (public/style.css):**

    Change #app and #game-container from centering layout to a flex-column layout where the canvas is on top and the text input bar is below.

    The #app div should be a column flex container. The #game-container should NOT be 100vh anymore -- it should take available space with flex:1 and max-height to leave room for the input bar.

    Add styles for #text-parser-ui, #parser-response, #parser-input matching the pixel art dark theme from the research (dark background #1a1a2e, monospace font, etc.). The input bar width should match the canvas width.

    Key CSS considerations:
    - The Phaser Scale.FIT mode may resize the canvas. The input bar should be inside #game-container so it scales together, OR it should be a sibling that matches width.
    - Use the simpler approach: place #text-parser-ui as a child of #game-container, positioned after the canvas. Use CSS to ensure it sits below the canvas.
    - Set max-width on #game-container to prevent the input bar from being wider than the game at large viewports.
