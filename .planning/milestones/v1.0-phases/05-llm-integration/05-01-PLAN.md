  <files>src/game/llm/OllamaClient.ts</files>
  <action>
Create `src/game/llm/OllamaClient.ts` -- a minimal HTTP wrapper around Ollama's `/api/chat` REST endpoint. Use raw `fetch()` (NOT the ollama npm package -- it adds 15KB for wrapping a single fetch call).

Constructor parameters:
- `baseUrl` (default `'http://localhost:11434'`)
- `model` (default `'qwen2.5:3b'`)
- `timeoutMs` (default `2000`)

Internal state:
- `available: boolean | null` (null = unchecked)
- `pendingController: AbortController | null` (for aborting in-flight requests)

Methods:
1. `async checkAvailability(): Promise<boolean>` -- GET to `this.baseUrl` with `AbortSignal.timeout(1000)`. Sets `this.available` to `res.ok` on success, `false` on catch. Returns the result.

2. `isAvailable(): boolean` -- Returns `this.available === true`. Synchronous check used by HybridParser before attempting LLM call.

3. `async chat(systemPrompt: string, userMessage: string, schema: object): Promise<{ content: string }>` -- POST to `${this.baseUrl}/api/chat` with body:
   ```
   {
     model: this.model,
     messages: [{ role: 'system', content: systemPrompt }, { role: 'user', content: userMessage }],
     format: schema,
     stream: false,
     options: { temperature: 0 },
     keep_alive: '30m'
   }
   ```
   Before sending: if `this.pendingController` exists, call `.abort()` on it (cancels previous in-flight request to prevent race conditions on rapid input). Create new `AbortController`. Combine it with `AbortSignal.timeout(this.timeoutMs)` via `AbortSignal.any([...])` for the fetch signal.
   On success: set `this.pendingController = null`, return `{ content: response.message.content }` from the JSON body.
   On non-OK response: throw `Error('Ollama returned ${res.status}')`.

4. `async warmUp(): Promise<void>` -- Fire-and-forget POST to `/api/chat` with a trivial message (`{ role: 'user', content: 'hi' }`), `stream: false`, `keep_alive: '30m'`. Use `AbortSignal.timeout(15000)` (longer timeout for model loading). Catch all errors silently -- warm-up failure is non-critical.

Export the class and the interfaces `OllamaChatRequest` and `OllamaChatResponse` (for type documentation, not runtime use).

IMPORTANT: Do NOT add `fetch` polyfills or imports. This runs in the browser where `fetch`, `AbortSignal.timeout()`, and `AbortSignal.any()` are all natively available.

**PromptBuilder** (`src/game/llm/PromptBuilder.ts`):

Create a class with a single `build()` method that assembles the system prompt and user message for the LLM.

Static readonly `SYSTEM_PROMPT` string that tells the LLM:
- It is a text adventure game command parser
- It must respond with JSON containing `verb`, `subject`, `target`
- Valid verbs: look, take, use, go, talk, open, push, pull, inventory, combine
- Rules for mapping intent to verbs (examine->look, pick up->take, movement->go with exit directions, "use X on Y"->use with both subject and target)
- Use object IDs from context when possible
- Return null for subject/target when not applicable

`build(input, hotspots, exits, inventoryItems, roomContext)` method:
- Parameters: `input: string`, `hotspots: Array<{ id: string; name: string }>`, `exits: Array<{ id: string; direction?: string; label?: string }>`, `inventoryItems: Array<{ id: string; name: string }>`, `roomContext: { name: string; description?: string }`
- Returns `{ systemPrompt: string; userMessage: string }`
- User message format:
  ```
  Room: {roomContext.name}
  Description: {roomContext.description}  (only if truthy)

  Objects here:
    - "{name}" (id: {id})
    ...or (none)

  Exits:
    - {direction || label || id} (id: {id})
    ...or (none)

  Inventory:
    - "{name}" (id: {id})
    ...or (empty)

  Player input: "{input}"
  ```
- Keep the prompt minimal -- do NOT include puzzle logic, zone coordinates, death triggers, or walkable areas. Only what the LLM needs to interpret the command.

**ResponseMapper** (`src/game/llm/ResponseMapper.ts`):

Export `GAME_ACTION_SCHEMA` constant -- the JSON schema object used as Ollama's `format` parameter:
```typescript
{
  type: "object",
  properties: {
    verb: { type: "string", enum: ["look","take","use","go","talk","open","push","pull","inventory","combine"] },
    subject: { type: ["string", "null"] },
    target: { type: ["string", "null"] }
  },
  required: ["verb", "subject", "target"]
}
```

Create a `ResponseMapper` class with:
- Private `nounResolver: NounResolver` instance (import from `../parser/NounResolver.ts`)
- `VALID_VERBS: Set<string>` with all valid verb strings

`map(responseContent, hotspots, exits, inventoryItems, rawInput)` method:
- `responseContent: string` -- the raw JSON string from the LLM's `message.content`
- Other params match existing types (HotspotData[], ExitData[], etc.)
- Returns `ParseResult`

Logic:
1. `JSON.parse(responseContent)` -- if it throws, return `{ success: false, error: "I didn't quite understand that." }`
2. Validate `parsed.verb` is in `VALID_VERBS` -- if not, return error ParseResult
3. Cast `parsed.verb as Verb`
4. If `parsed.subject` is truthy string, resolve it through `this.nounResolver.resolve(parsed.subject, hotspots, exits, inventoryItems)` and use `.id`
5. If `parsed.target` is truthy string, resolve it the same way
6. Build and return `{ success: true, action: { verb, subject, target, rawInput } }`

Import types from `../types/GameAction.ts` (ParseResult, Verb, GameAction) and `../types/RoomData.ts` (HotspotData, ExitData).

IMPORTANT: The NounResolver import path is `../parser/NounResolver.ts`. Do NOT create a new noun resolver -- reuse the existing one which already handles exact ID, exact name, partial word matching against hotspots, exits, and inventory items.
