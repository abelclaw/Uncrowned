**New imports:**

import { PuzzleEngine } from './PuzzleEngine';

**New constructor (was stateless, now holds references):**
```typescript
class CommandDispatcher {
    private puzzleEngine: PuzzleEngine;
    private state: GameState;
    private itemDefs: ItemDefinition[];

    constructor(itemDefs: ItemDefinition[] = []) {
        this.puzzleEngine = new PuzzleEngine();
        this.state = GameState.getInstance();
        this.itemDefs = itemDefs;
    }
}
```

**Revised dispatch() method -- evaluation order is CRITICAL:**
1. Handle meta-commands first (no room interaction): 'inventory', 'save', 'load'
2. Check death triggers (roomData.deathTriggers) via puzzleEngine.tryPuzzle -- if matched, return response (death event already emitted by PuzzleEngine)
3. Check puzzle interactions (roomData.puzzles) via puzzleEngine.tryPuzzle -- if matched, return response
4. Fall through to enhanced verb handlers (existing handlers, now inventory-aware)

**New verb handlers:**
- `handleInventory()`: Return a formatted list of inventory item names from GameState. If empty, return "Your pockets are empty. How tragic." Also emit 'inventory-toggle' event via EventBus so the InventoryPanel can toggle.
- `handleSave(action)`: Extract slot number from action.subject (parsed from "save 1"). If no slot, return "Save to which slot? (1-5)". Call SaveManager.saveToSlot(). Return "Game saved to slot {N}."
- `handleLoad(action)`: Extract slot number from action.subject. If no slot, list available slots via SaveManager.getSlotInfos(). Call SaveManager.loadFromSlot(). If loaded, emit 'load-game' event via EventBus with roomId. Return "Game loaded." If slot empty, return "No save in slot {N}."
- `handleCombine(action)`: Check if both subject and target are in inventory. Look up combination puzzle in a global puzzles list or room puzzles. If matched, execute via PuzzleEngine. If no combination exists, return "Those don't go together."

**Enhanced existing handlers:**
- `handleLook`: Before returning room description, check roomData.dynamicDescriptions -- iterate flag keys, if GameState has that flag set, use the dynamic description instead. When looking at a subject, check inventory items (via itemDefs) AFTER checking hotspots -- if subject matches an inventory item the player HAS, return item.description from itemDefs.
- `handleTake`: Before returning "can't take", check roomData.items for matching item. If found and not already taken (check GameState.isRoomItemRemoved), run through puzzleEngine for matching take puzzle. If no puzzle match, do a simple add-item + narrator response.
- `handleUse`: For two-noun "use X on Y", resolve X as inventory-first (check GameState.hasItem), resolve Y as room-first (check hotspots). If X is inventory item, check puzzles for "use X on Y" match. If single-noun "use X", check room hotspots first, then inventory items.

**findItem helper:**
Add `findItem(subject: string): ItemDefinition | undefined` that searches itemDefs by ID, name, and partial name match (same pattern as findHotspot).

**Key design decisions (from research):**
- Death triggers checked BEFORE puzzles (research Pitfall #3 says check puzzles first to avoid false deaths, but the research ALSO says to check death triggers with MORE specific conditions). Resolution: Check puzzles first (solutions take priority), THEN death triggers. A successful puzzle match short-circuits death trigger evaluation. This prevents "player has correct items but still dies."
- "use X on Y": X resolves inventory-first, Y resolves room-first (research Pitfall #4 and Open Question #3)
- Items only consumed by explicit puzzle actions, NEVER by failed attempts (research anti-pattern: "never consume key items on wrong actions")

Update: Actually re-reading the research more carefully, the order should be:
1. Meta-commands (inventory, save, load)
2. Puzzles (solutions -- these should succeed if conditions are met)
3. Death triggers (only if no puzzle matched -- prevents dying when you have the solution)
4. Standard verb handlers (static responses)

This matches research Pitfall #3: "check puzzles first (solutions), then death triggers."
